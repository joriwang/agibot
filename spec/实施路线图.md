# 五阶段路线图

## 阶段一：建立 Specs 基础设施（不动现有代码）✅ 已完成

**目的**：为现有代码编写完整的逆向规格说明，建立"单一事实来源"。

**内容**：
- L0 架构 Spec（1 份）——系统全貌、模块关系、数据流、线程模型、依赖清单
- L1 模块 Spec（4 份）——每个模块的接口契约、内部逻辑、状态机、错误处理
- L2 协议 Spec（6 份）——所有消息类型的字段语义、生产者/消费者、取值约束
- 附带产出：bugs.md（7 个 bug）、交叉索引、测试标准、版本绑定规范

**验收标准**：新人读完 spec 能理解系统全貌；AI 读完 spec 有足够信息生成代码。

**当前状态**：圆满完成，spec 质量远超初稿预期。

---

## 阶段二：在"安全区"试点 Spec-Driven 生成 ← 当前位置

**目的**：用一个风险最低的模块验证"从 spec 生成代码"的可行性，建立流程和信心。

**内容**：
- 选定试点模块（已确定：SimModule）
- 设计 prompt 策略和材料组合
- AI 从 spec 生成代码，与现有代码 diff 对比
- 量化评估：直接可用率、需人工干预的点
- 沉淀可复制的生成流程模板

**验收标准**：生成代码能编译通过、接口兼容、通过 Test Criteria；产出一套可复制的 prompt + 评估流程。

---

## 阶段三：中间层模块逐个迁移

**目的**：将 spec-driven 生成扩展到更复杂的业务模块，逐步提高覆盖率。

**内容**：
- 按依赖拓扑从简到难：JoyStickModule → ControlModule
- 每个模块走完"spec → 生成 → diff → 验证"闭环
- 迭代优化 spec 模板和 prompt 策略（用阶段二的经验反哺）
- 建立模块级的回归测试基线

**验收标准**：越来越多的模块可以从 spec 重新生成，人工干预率逐步降低。

---

## 阶段四：核心层增强约束式管理

**目的**：对硬件驱动等不适合完全自动生成的模块，用 spec 做约束文档和审查依据。

**内容**：
- DcuDriverModule 不强求从 spec 生成（硬件耦合太重）
- 但任何代码修改都必须先更新 spec，再改代码
- CI 检查 spec 与代码的一致性（至少接口层面）
- spec 作为 code review 的审查基准

**验收标准**：即使不能自动生成，也实现了 spec-first 的开发纪律。代码变更必有 spec 变更。

---

## 阶段五：搭建自动化管线

**目的**：将前四个阶段的手动流程固化为 CI/CD 自动化流程。

**内容**：
- spec 变更 → 触发 AI 代码生成 → 自动编译 → 自动运行 Test Criteria → 人工审核
- 版本管理自动化：`specs_vN.md` ↔ `code_vN/` 对应关系
- 可选：搭建类似 OnlySpecs 的本地工具管理整套流程
- spec 过期检测：代码变更但 spec 未更新时自动告警

**验收标准**：团队日常开发中，spec 更新 → 代码生成 → 测试验证形成半自动闭环。
